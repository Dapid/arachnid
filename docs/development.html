Development¶

See Arachnid API for code documentation.

Getting Source¶

To get the latest version of the source, use the following git command:

$ git clone http://github.com/ezralanglois/arachnid

Alternatively, you can download a zip file of the latest development release:

Download

Every time you update C or Cython code, you may run the following command (rather that reinstall)

$ python setup.py build_ext --inplace

Every time you add a new script or the first time you get the source, you may run the following command (rather that reinstall)

$ python setup.py develop --install-dir $BIN -m

Reporting Bugs¶

As a user or developer, you can report bugs or request features on the Issues Tracker. Please, search for your issue before submitting it.

Contributing Code¶

The preferred way to contribute to Arachnid is to fork the main repository on Github and submit a “Pull Request”.

  1. Create a GitHub account

  2. Fork the Arachnid repository

  3. Clone a local copy to your machine

    $ git clone git@github.com:<YourLogin>/arachnid/arachnid.git

    Replace <YourLogin> with your username for GitHub.

  4. Create a branch to hold your changes

    $ git checkout -b my-changes

  5. When you have finished making changes, then

    $ git add modified_file1
$ git add modified_file2
$ git commit -m "Comment describing your modifications"

  6. Upload your changes to your fork of Arachnid on GitHub

    $ git push -u origin my-feature

  7. Make a request to incorporate your changes

    Go your fork of Arachnid on Github and click the “Pull request” button. This sends your changes to the maintainers for review.

Your code should be error free and conform to the current code (also avoid import *). You can use the following tools to help ensure your code conforms to the proper standards.

  • Basic documentation and error checking

    $ easy_install pyflakes
$ pyflakes path/to/module.py

  • Unit test coverage

    $ easy_install nose coverage
$ nosetests --with-coverage path/to/tests_for_package

Debugging C/C++ code¶

Memory errors are the bain of every C/C++ developers existence. One important tool to finding these errors is valgrind.

  1. Install valgrind
  2. Download and modify the Python suppressions file: valgrind-python.supp See the directions in the README.valgrind for more information on modifying the suppressions.
  3. Run valgrind over your code (choose a small example because there is a performance cot)
$ valgrind -v --suppressions=valgrind-python.supp python my_test_script.py

Testing compilation of code¶

To test whether your code has any problems such as uninitalized variable, use the following flags in the environment for GCC type compilers.

$ export CFLAGS="-Werror -Wno-unused-function -Wno-unknown-pragmas -Wno-format" CXXFLAGS="-Werror -Wno-format -Wno-unknown-pragmas -Wno-unused-function"
$ export FFLAGS="-Werror -Wno-unused-function -Wtabs" F90FLAGS="-Werror -Wno-unused-function -Wtabs" F77FLAGS="-Werror -Wno-unused-function -Wtabs"

Packaging for Anaconda¶

Build System: CentOS 5.10¶

The Anaconda people use CentOS 5 to build Anaconda and to ensure compatibility I recommend doing the same.

Note

Redhat backported OpenMP to gcc 4.1. This provides both backward compatibility for gcc4 compilers and OpenMP threading.

Here are the steps I suggest to perform a minimal install of CentOS 5 on some platform, e.g. virutual machine.

  1. Get the ISO.

You can download CentOS 5.10 from http://isoredirect.centos.org/centos/5/isos/x86_64/

I am going through a bare minimum build so I only use the first CD: CentOS-5.10-x86_64-bin-1of9.iso

  1. Install

To do a bare minimum install, I recommend unselecting all the packages. This means selecting the Customize Now option when you see it and unselecting the Gnome Desktop.

Warning

Do not use automatic installation in VMWare if you want a bare minimum install of CentOS 5.

  1. Install Compilers

    $ yum install gcc gcc-gfortran gcc-c++ -yq

  2. Install Git Prerequisites

    1. Install Git
    $ yum install wget -yq
$ yum install unzip -yq
$ yum install make -yq
$ yum install zlib zlib-devel -yq
$ yum install openssh-server openssh-clients openssl-devel -yq
$ yum install curl curl-devel -yq
$ yum install expat expat-devel
$ yum install gettext
$ yum install bzip2

  1. Install Git

    $ wget https://github.com/git/git/archive/master.zip
$ mv master master.zip
$ unzip master.zip
$ cd git-master/
$ make prefix=/usr install

  1. Install Anaconda

    $ wget http://repo.continuum.io/miniconda/Miniconda-3.0.0-Linux-x86_64.sh
$ sh Miniconda-3.0.0-Linux-x86_64.sh -b -p $PWD/anaconda

  2. Get Arachnid Source

    $ git clone http://github.com/ezralanglois/arachnid/

Documentation Hack¶

To get the documentation to build correctly, you need to edit sphinx/ext/autosummary/generate.py in your site-packages directory.

Change Line 143 from

for name in dir(obj):

to

for name in vars(obj):

A little background: The default autosummary code gets all inherited members of a class. This ensures only the current members will be documented.

Create a Program Script¶

Arachnid supports two types of scripting: applications and fast prototyping. Applications are scripts that are designed for a user and to fit on the reconstruction workflow.

Basic Program¶

Essentially a script can utilize one of two program architectures:

  1. Batch processing
  2. Group or file processing

This is determined by whether the script defines a process() or batch() entry point.

By convention, a program should define a main entry point for execution from outside the script as follows:

def main():
    #Main entry point for this script
    from arachnid.core.app import program
    program.run_hybrid_program(__name__, description = """Description of the script""")

if __name__ == "__main__": main()

Basically, it should define a main function that can serve as an entry point for setup.py. And, in case the script is executed from the command line, it should contain the following:

if __name__ == "__main__": main()

Interface¶

The target module must define one of the following functions:

process(filename, **extra)¶

Process a single file. During parallel processing this is invoked by a worker process.

See arachnid.core.app.file_processor for more information on the file or group processing architecture.

Parameters:
  • filename – Input filename to process
  • extra – Unused keyword arguments (Options from the command line or config file, plus additional options)
Returns:

At minimum filename, however a tuple of additional arguments can be returned for processing by reduce_all()
batch(files, **extra)¶

Process all input files.

Parameters:
  • files – List of input filenames to process
  • extra – Unused keyword arguments (Options from the command line or config file, plus additional options)

The target module must define all the the following:

setup_options(parser, pgroup=None, main_option=False)¶

Setup program options for command-line and configuration file

Parameters:
  • parser – OptionParser
  • pgroup – Root OptionGroup
  • main_option – If true, add the options specific to running this script

The target module may include any of the following functions:

change_option_defaults(parser)¶

Customize default options in other modules for this specific script.

Parameters:parser – OptionParser
dependents()¶

Return a list of dependent modules and add their options. Dependents of a dependent will also be recursively found and added.

Returns:List of dependent modules
update_options(options)¶

Utilize parsed options and add additional option values.

Parameters:options – object containing options as fields
check_options(options, main_option=False)¶

Test the validity of each option value and throw OptionValueError if an invalid value is found.

Parameters:
  • options – object containing options as fields
  • main_option – If true, add the options specific to running this script
flags()¶

Return a dictionary of support properties

Returns:Dictionary of supported properties

File Processor¶

The file processor module provides basic functionality to any program that processes a set of files or a set of object groups independently. It can distribute processing to multiple cores using the multiprocessing package as well as to multiple nodes on a cluster using mpi2py.

The target module must define the following functions:

process(filename, **extra)

Process a single file. During parallel processing this is invoked by a worker process.

Parameters:
  • filename – Input filename to process
  • extra – Unused keyword arguments (Options from the command line or config file, plus additional options)
Returns:

At minimum filename, however a tuple of additional arguments can be returned for processing by reduce_all()

The target module may include any of the following functions:

initialize(files, extra)¶

Initialize the input data before processing. During parallel processing this is invoked by all processes.

Parameters:
  • files – List of input files to process
  • extra – Dictionary of unused keyword arguments (Options from the command line or config file, plus additional options)
Returns:

None or a new list of files or objects to be processed by other routines
finalize(files, **extra)¶

Finalize data after processing. During parallel processing this is invoked by all processes.

Parameters:
  • files – List of input files to process (possibly modified by initialize())
  • extra – Unused keyword arguments (Options from the command line or config file, plus additional options)
reduce_all(filename, file_index=<int>, file_count=<int>, file_completed=<int>, **extra)¶

Reduce data from worker to root process. During parallel processing this is invoked by the root process. Note that only filename is a positional argument.

Parameters:
  • filename – Input filename that was processed. May be a tuple containing data resulting from process().
  • file_index – Current index in the list of input filenames
  • file_count – Number of input filenames
  • file_completed – Current count of finished files
  • extra – Unused keyword arguments (Options from the command line or config file, plus additional options)
Returns:

Only the filename that was processed

Options¶

Adding an option to the Improved OptionParser can be done simply as follows:

group.add_option("",   invert_contrast=False,        help="Invert the contrast of CCD micrographs")

The first argument to add_option can be an empty string, a short parameter (-i) or a long parameter (–invert-image-contrast).

The second argument to add_option defines several values simutaneously

  1. dest: The name of the destination in the values object

  2. default: The default value of the option, e.g. False for this boolen option

  3. type: The expected type of the input argument

  4. action: For boolean types, it sets the action to store_false or store_true depending on the initial value

  5. long form argument: The long form argument in the configuration file or on the command

    line, e.g. invert_contrast becomes –invert-contrast

The third argument in this example defines a help message as in the standard OptionParser.

Note

The arguments expected by the original OptionParser can also be passed to add_option

The add_option function supports several additional parameters including:

  1. dependent: Mark option such that changing its value does not force a restart in processing all files
  2. required: Mark option as required, i.e. if the option is not given a value, then it will throw an error
  3. required_file: Marks are required and tests if filename is directory, then throws an error if so

The add_option function can optionally take gui parameter that defines type specific information for building a GUI component of that option.

Consider some examples:

A list of input files would have the following example signature:

group.add_option("-i", input_files=[], help="List of filenames for the input micrographs", required_file=True, gui=dict(filetype="file-list"))

A non-required input file would have the following example signature:

group.add_option("-d", selection_doc="",       help="Selection file for a subset of micrographs or selection file template for subset of good particles", gui=dict(filetype="open"), required_file=False)

An output file would have the following example signature:

group.add_option("-o", output="",      help="Output filename for the coordinate file with correct number of digits (e.g. sndc_0000.spi)", gui=dict(filetype="save"), required_file=True)

An integer valued option would have the following example signature:

group.add_option("",   limit=2000,      help="Limit on number of particles, 0 means give all", gui=dict(minimum=0, singleStep=1))

An real valued option would have the following example signature:

group.add_option("",   disk_mult=0.65,      help="Disk smooth kernel size factor", gui=dict(maximum=10.0, minimum=0.01, singleStep=0.1, decimals=2))

Logging¶

Standard practice when using this module is to define a logger for each module as follows:

import logging
_logger = logging.getLogger(__name__)
_logger.setLevel(logging.DEBUG)

Note that you do not need to import this module, only the standard logging library.

On the next line, we define a logger for this module called _logger. Passing __name__ to getLogger creates a logger with a name based on the module.

On the following line, we set the default log level. If you want debug logging to print in -v4 then set the level to logging.DEBUG. However, if you want to suppress debug logging until the user specifies -v5 then set the level to logging.INFO.

A message can bypass the standard logger and go directly to the crash report by setting the tofile to True. For example, consider the following:

logging.warn("Some message", extra=dict(tofile=True))

Logging failed imports of optional modules should be done as follows:

try:
    from util import _module
    _module;
except:
    from arachnid.core.app import tracing
    tracing.log_import_error('Failed to load _module.so module - certain functions will not be available', _logger)
    _module = None

This generates a list of failed imports of optional modules, which will be logged at the start of the crash report.

Basic Snippets¶

Fast prototyping is a way to test new ideas, perform a simple task quickly or customize objects such as plots.

The snippets cover a set of examples for writing fast-prototype code. These examples range from unstacking a SPIDER image stack to customizing the FSC plot.

Below are some very short snippets illustrating basic utilities in Arachnid.

Reading/Writing images¶

This module supports the standard formats for image data in cryo-EM, which include:

  • MRC
  • SPIDER
  • EMAN2/SPARX (Optional, used if available)

Note

Each read function supports soft links or shortcuts

Images are read in as standard NumPy arrays using C-ordering for 2D or 3D images.

from arachnid.core.image import ndimage_file as imfile
image = imfile.read_image('image.spi')

print 'Mean pixel:', image.mean()
print 'Image size (rows, columns):', image.shape

Due to the sheer number, images in cryo-EM are often grouped into stacks. The number of images in a stack can be determined by:

print "Number of images:", imfile.count_images('stack.spi')

A NumPy array containing pixel values can be written out as an image using the following:

imfile.write_image('image.mrc', image)

Reading/Writing metadata¶

The metadata format module supports reading and writing metadata in a number of document formats including:

  • Comma Separated Value Format

    arachnid.core.metadata.formats.csv

  • Malibu Prediction Format

    arachnid.core.metadata.formats.prediction

  • Spider Document Format

    arachnid.core.metadata.formats.spiderdoc

  • Spider Selection Format

    arachnid.core.metadata.formats.spidersel

  • STAR Format

    arachnid.core.metadata.formats.star

Metadata can be read in using the following command:

>>> from arachnid.core.metadata import format

>>> data = format.read('sndc_0000.dat')

By default, the data is a list of NamedTuples

>>> print data
[ BasicTuple(id="1/1", select=1, peak=0.00025182), BasicTuple(id="1/2", select=1, peak=0.00023578) ]

Note

The format of a document will be automatically determined when the document is read.

The data can be alternatively stored as a NumPy array, but the header is returned as a separate field:

>>> data_array, data_header = format.read('sndc_0000.dat', ndarray=True)

Similarly, the data (list of Namedtuples) can be written out using the following:

>>> format.write("outputfile.star", data)

Note

When writing out a document, the format will be chosen based on the extension.

If the data is in a NumPy array, then the header must be added as an additional argument:

>>> header = ['id', 'select', 'peak']
>>> format.write("outputfile.star", data, header=header)
